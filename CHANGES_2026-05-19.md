## What shipped

### 1. Shared billing-address modal (new include)

**File**: `_includes/billing-address-modal.html`

Reusable modal anywhere the site needs to capture or edit an account's billing address. Single global overlay (singleton-guarded on init) that any page can open via:

```js
window.eeBillingAddress.open({
    accountId: 50430539,
    currentAddress: {...} || null,
    reason: 'profile' | 'invoice' | 'quote'
}).then(savedAddress => { ... });
```

- Country dropdown ships the spec's top-of-list (CA, US, GB, AU, DE, NL, NZ, FR) followed by a curated long-tail.
- Province / state dropdown auto-swaps:
  - **CA** → 13-province dropdown, required.
  - **US** → 50 states + DC, required.
  - **Other** → free-text input, optional (matches backend gate which only requires `state` for CA/US).
- Postal code is required for CA + US, optional otherwise.
- Client-side validation mirrors the backend errors documented in §1 of the integration doc.
- POSTs to `/set_billing_address` itself; on success invalidates the shared `window.eeDashboard` cache so subsequent reads see the new address.
- Dispatches a `billingAddressUpdated` DOM event so any open page can refresh its display without polling.
- Resolves the promise with the saved address on success, `null` on cancel (× / backdrop / Escape / Cancel button).

### 2. Billing-address card on /account/billing/ (Purchase history)

**File**: `billing.html`

New card at the top of the page (above Payment method, above Invoices & receipts):

- Reads `account.address` from the dashboard (new field shipped per the integration doc).
- Renders a multi-line "envelope address block" when set, or *"No billing address on file. Add one before requesting an invoice or quote."* placeholder when null.
- **Edit** button opens the modal (`Add address` label when none yet, `Edit` once present).
- Listens for the global `billingAddressUpdated` event + patches in-memory state on save so the card refreshes instantly without a dashboard re-fetch.
- Account-switch (page-title dropdown → `accountScopeChanged`) re-renders the card alongside the timeline.

### 3. 409 `billing_address_required` retry on purchase page

**File**: `purchase.html`

Both `onInvoice` and `onQuote` now:

- Catch the structured 409 (`err.status === 409 && err.data.code === 'billing_address_required'`).
- Open the shared modal with `reason: 'invoice'` or `'quote'` (which adjusts the modal's title + intro copy to explain why the address is suddenly required).
- On save, patch the in-memory `state.accounts[…].address` and retry the original request exactly once.
- On user cancel (modal closed without save) bail silently — no error toast.

Matches the recommended pattern from §3 of the integration doc.

### 4. Structured custom fields (replaces PO + Attention)

**File**: `purchase.html` (invoice details panel)

- The previous two static text inputs (PO Number, Attention) are now a dynamic list of `{name, value}` rows, capped at 4 per Stripe's invoice limit.
- Two rows are pre-seeded with the canonical names from §4 of the doc: **PO Number** and **Reference**. Names are editable so customers can rename (e.g., "Cost Center", "Department", "Solicitation Number").
- **+ Add another field** button appends a row; **×** removes a row.
- All inputs `maxlength="30"` (matches the Stripe constraint).
- Empty rows are filtered out before submit. `custom_fields` key is omitted entirely from the request body when no rows are populated — avoids the backend's "non-empty name and value" validator.
- On the request body, sent as the documented `custom_fields: [{name, value}, ...]` structure for both invoice and quote endpoints. Backwards-compat detritus from the old `footer`/`po_number`/`attention` shape was removed.

### 5. Per-license-card UX restyle on Licenses & credits

**File**: `account.html`

Per the user-approved design from yesterday:

- Card layout switched to a two-column flex row: title + meta + relationship on the left, action buttons stacked on the right.
- Each action button has a primary label + small subtitle:
  - **Renew for 1 year** — *"Extends this license. Starts at expiry."*
  - **+ Add devices** — *"Prorated to current expiry."*
- Outline-blue style; hover fills the button.
- Below 600px viewport, columns collapse so buttons stack below the info (mobile-friendly).
- Foldout details panel + bottom chevron toggle still span the full card width unchanged.

## What's deferred (not in this commit)

These pieces from the integration doc are intentionally not implemented yet — the highest-value parts are above, and the rest are scoped follow-ons:

### `renew_license` endpoint integration (purchase.html)

The doc adds a `renew_license` endpoint with `payment_method` modes (`checkout` / `invoice` / `quote`). The purchase page has the *markup* for a Renew mode (`#modeRowRenew`) but the JS plumbing is incomplete:
- `state.renewLicenseId` state field exists but isn't yet populated by clicks.
- `renderRenewLicenseList()` doesn't exist yet.
- `onContinue` dispatcher doesn't branch on `state.mode === 'renew'`.
- The "Renew for 1 year" button on each license card still opens a mailto, not the purchase page.

**Next step**: complete the renew-mode wiring, then change the per-card Renew button's `href` from `mailto:…` to `/account/purchase/?mode=renew&license_id=X`. The renew endpoint already supports invoice / quote payment methods + custom_fields + the 409 address gate, so the existing retry logic on `purchase.html` will work as-is.

### `quote.accepted` polling on /account/billing/

§5 of the doc notes there's no webhook handler yet for quotes the customer self-accepts on Stripe's hosted page. Recommended action is for the frontend to re-poll `list_account_billing` after the customer accepts, but we have no signal in the UI for "I just accepted a quote." A natural place to add a polling loop is post-accept-success, but since we don't currently surface a Quote → Invoice step in the customer UI (only admin staff would convert via `admin_accept_quote`, which we also haven't built UI for), I left this out. When the customer-facing accept flow lands, add a 5-second poll for 60 seconds after the new tab opens, with `list_account_billing` looking for the new invoice.

### `admin_accept_quote` admin tool

§5 mentions a staff-only "Convert to Invoice" button on quote rows. We don't have an admin tooling page on this branch (the impersonation form is the only admin surface). Defer until we add an admin tab.

### Address field on Account access view

`accounts[i].address` is exposed by the dashboard and any owner can edit it. I put the Edit UI on /account/billing/ (Purchase history) since that page is the existing billing-affordance home. An argument can be made for adding the field to Account access too — it's a per-account setting, like the owners list. Left for a separate decision.

### Tax-line surfacing on the timeline

The `list_account_billing` payload doesn't currently break out tax separately on the invoice / charge rows we render. Stripe's hosted invoice page shows the breakdown when the customer pays, but if you want the timeline to render "subtotal + tax + total" inline, the backend would need to expose the breakdown fields on each item. Not blocking; flag for backend.
